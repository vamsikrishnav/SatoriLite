File System Access API

Last updated: 2026-05-28 • 8 min read

Overview

The File System Access API enables web applications to read and write files on the user's local filesystem. Unlike older approaches (input type="file", download attribute), this API provides persistent, read-write access to directories the user grants permission to.

SatoriLite uses this API as its primary storage mechanism — every note is a real .md file on disk, accessible by git, VS Code, or any other tool.

Info

This API requires a secure context (HTTPS or localhost) and user activation (a click or keypress) to trigger the file picker.

Browser Support

The File System Access API is currently supported in Chromium-based browsers only:

  • Chrome 86+ — Full support
  • Edge 86+ — Full support
  • Arc — Full support (Chromium-based)
  • Brave — Full support (Chromium-based)
  • Firefox — Not supported (no plans announced)
  • Safari — Not supported
Warning

SatoriLite deliberately targets Chromium-only. We chose real filesystem access over cross-browser compatibility because the alternative (IndexedDB/OPFS) traps files in a browser sandbox where "Clear Browsing Data" destroys everything.

API Usage

The core workflow involves three steps: opening a directory, reading files, and writing changes back.

Opening a Directory

const dirHandle = await window.showDirectoryPicker({ id: 'satori-vault', mode: 'readwrite', startIn: 'documents' }); // Store handle for persistence across sessions const db = await openDB('satori', 1); await db.put('handles', dirHandle, 'vault');

Reading a File

async function readFile(dirHandle, path) { const parts = path.split('/'); let current = dirHandle; for (const part of parts.slice(0, -1)) { current = await current.getDirectoryHandle(part); } const fileHandle = await current.getFileHandle(parts.at(-1)); const file = await fileHandle.getFile(); return file.text(); }

Writing a File

async function writeFile(fileHandle, content) { const writable = await fileHandle.createWritable(); await writable.write(content); await writable.close(); }

Security Model

The API enforces a permission model that balances usability with safety:

  • User activation required — file picker can only be shown in response to a user gesture
  • Per-origin permissions — each origin must be granted access separately
  • Persistent permissions — permissions survive browser restarts (until revoked)
  • Sensitive directories blocked — system directories, browser profiles, and other protected paths are off-limits

The browser treats directory handles like capabilities — possessing one is proof of user consent.

Limitations

Tip

Despite these limitations, the File System Access API is the most capable filesystem interface available to web apps today. For a personal wiki that values data ownership, it's the right trade-off.

  • No watch API — can't detect external file changes without polling
  • Permission prompts — browser may re-prompt after extended inactivity
  • No symlinks — API doesn't expose or follow symbolic links
  • File size — very large files (100MB+) can cause memory pressure since the entire file is read into memory
  • Concurrency — no built-in locking mechanism for concurrent writes from multiple tabs
